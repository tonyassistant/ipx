# Network discovery

ipx builds its view from native operating-system networking commands.

## Current discovery sources

### macOS

- `networksetup -listallhardwareports`
- `networksetup -listnetworkserviceorder`
- `ifconfig`
- `netstat -rn -f inet`

### Linux

- `ip -o link show`
- `ip -o -4 addr show`
- `ip route show default`

### Windows

- `ipconfig /all`

## What this enables

ipx can currently map:

- interfaces and devices
- basic addressing and status information
- default gateway and default-route context
- virtual interfaces on supported platforms
- service-to-interface mapping where the platform exposes it clearly, including primary-service priority on macOS

## Platform notes

- macOS currently has the richest service mapping
- Linux currently focuses on interface, address, gateway, and virtual-interface discovery
- Windows currently focuses on interface, address, gateway, and adapter classification

## Reachability posture

ipx derives an inspect-first reachability posture from address and route data.

Current states:
- `reachable` for interfaces that appear globally routed
- `private route` for interfaces that carry a default route on private upstream space
- `local only` for interfaces with local addressing but no credible upstream path
- `down` for disconnected interfaces
- `unknown` for inactive interfaces

This is not a packet probe yet. It is a conservative operator hint based on discovered route truth, and it now keeps CGNAT, documentation/test ranges, multicast, and other reserved IPv4 space out of the public-upstream bucket.

## v1 expectation

This is enough for a strong inspect-first v1, while deeper signal, service-state, probe-backed validation, and platform-parity work continues over time.
